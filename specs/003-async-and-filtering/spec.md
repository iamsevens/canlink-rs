**版本**: 1.0.0

**状态**: 草案
**优先级**: P1

## 概述

本规范定义了 CANLink-RS v0.2.0 的两个核心功能：异步 API 支持和消息过滤功能。这些功能将提升库的易用性和性能，支持更复杂的应用场景。

### 目标

1. 提供可选的异步 API，支持 async/await 编程模型
2. 实现高效的消息过滤机制，减少不必要的消息处理
3. 完成 v0.1.0 延迟的 8 项功能改进

### US2: 消息过滤 (P1)

**作为** 一名只关心特定 CAN ID 的工程师
**我希望** 配置消息过滤器
**以便** 我只接收感兴趣的消息，减少处理开销

#### 验收场景

**场景 2.1**: 单 ID 过滤
```
Given 已配置的过滤器，只接受 ID=0x123
When 总线上有 ID=0x123 和 ID=0x456 的消息
Then 只有 ID=0x123 的消息被接收
```

**场景 2.2**: ID 范围过滤
```
Given 已配置的过滤器，接受 ID 范围 0x100-0x1FF
When 总线上有各种 ID 的消息
Then 只有 0x100-0x1FF 范围内的消息被接收
```

**场景 2.3**: 掩码过滤
```
Given 已配置的掩码过滤器 (ID=0x120, Mask=0x7F0)
When 总线上有各种 ID 的消息
Then 只有匹配掩码的消息被接收 (0x120-0x12F)
```

**场景 2.4**: 硬件加速过滤
```
Given 硬件支持过滤功能
When 我配置过滤器
Then 过滤应该在硬件层面执行
And 减少 CPU 负载
```

#### 优先级原因
消息过滤是 CAN 应用的基本需求，可以显著减少软件处理负载，特别是在高流量总线上。

#### 独立测试
可以使用 MockBackend 测试软件过滤逻辑，硬件过滤需要真实设备验证。

### US3: 系统健壮性改进 (P2)

**作为** 一名需要长时间运行 CAN 应用的工程师
**我希望** 系统能够自动处理各种异常情况
**以便** 我的应用可以稳定运行而无需人工干预

#### 验收场景

**场景 3.1**: 硬件断开检测
```
Given 正在运行的 CAN 后端
When 硬件设备被断开
Then 系统应该检测到断开
And 返回适当的错误
And 允许重新连接
```

**场景 3.2**: 消息队列满处理
```
Given 高流量的 CAN 总线
And 配置了队列溢出策略（DropOldest/DropNewest/Block）
When 接收队列已满
Then 系统按配置的策略处理溢出
And 通过日志记录丢弃事件
```

**场景 3.2a**: 队列溢出策略配置
```
Given 用户需要自定义队列溢出行为
When 用户通过配置文件或 API 设置 QueueOverflowPolicy
Then 系统使用用户指定的策略
And 未配置时使用默认策略 DropOldest
```

**场景 3.3**: 资源泄漏防护
```
Given 长时间运行的应用
When 反复打开和关闭通道
Then 不应该有内存泄漏
And 不应该有句柄泄漏
```

**场景 3.4**: 日志记录
```
Given 启用了日志功能
When 发生错误或重要事件
Then 应该记录详细的日志信息
And 支持不同的日志级别
```

#### 优先级原因
这些改进对于生产环境的稳定性至关重要，是 v0.1.0 延迟的功能。

#### 独立测试
可以通过模拟各种异常情况进行测试。

### 消息过滤 (FR-005 到 FR-009)

- **FR-005**: 提供 `MessageFilter` trait 定义过滤接口
- **FR-006**: 支持单 ID 过滤、ID 范围过滤、掩码过滤；多个过滤器使用 OR 逻辑（任一过滤器匹配即通过）
- **FR-007**: 支持硬件过滤（如果硬件支持）
- **FR-008**: 支持软件过滤（作为后备方案）
- **FR-009**: 过滤器可以动态添加和移除

### 系统健壮性 (FR-010 到 FR-017)

- **FR-010**: 检测硬件断开连接并返回适当错误
  - 提供可选的自动重连机制
  - 可配置重试次数和间隔
  - 默认不自动重连，用户可通过配置启用
- **FR-011**: 定义消息队列满时的处理策略：
  - 提供 `QueueOverflowPolicy` 枚举，支持用户配置
  - 策略选项：`DropOldest`（丢弃最旧消息）、`DropNewest`（丢弃最新消息）、`Block`（阻塞等待，带超时）
  - 默认策略：`DropOldest`（适合实时监控场景）
  - 通过配置文件或 API 参数设置
- **FR-012**: 实现资源泄漏检测机制
  - 不做运行时检测（避免性能开销）
  - 通过测试（valgrind/miri）和文档保证资源正确释放
  - 所有资源类型实现 `Drop` trait
- **FR-013**: 集成日志框架
  - 使用 `tracing` 框架（与 tokio 生态兼容）
  - 支持结构化日志和 span
  - 通过 feature flag 可选启用
- **FR-014**: 支持配置热重载
  - v0.2.0 实现基础热重载功能
  - 监听配置文件变化，自动重新加载
  - 热重载时保持现有连接，仅更新可热更新的参数
- **FR-015**: 定义后端切换时的状态迁移
  - 切换时丢弃所有未处理消息，干净切换
  - 用户应在切换前自行处理完队列中的消息
  - 切换后需要重新配置过滤器等状态
- **FR-016**: 定义高频消息时的性能降级策略
  - 仅记录警告日志，继续尽力处理
  - 不做自动采样或背压（CAN 总线无法背压）
  - 用户可通过日志监控并自行决定处理方式
- **FR-017**: 定义内存使用限制
  - 通过配置最大队列大小（消息数量）限制内存
  - 超过限制时按 `QueueOverflowPolicy` 处理
  - 默认队列大小：1000 条消息（可配置）

## 成功标准

### SC-001: 异步性能
- **指标**: 异步 API 的吞吐量不低于同步 API 的 95%
- **测量方法**: 使用相同的测试场景比较同步和异步 API 的消息吞吐量
- **成功标准**: 异步 API 吞吐量 ≥ 同步 API 吞吐量 × 0.95

### SC-002: 过滤效率
- **指标**: 硬件过滤减少 CPU 负载 50% 以上
- **测量方法**: 在高流量场景下比较有无硬件过滤的 CPU 使用率
- **成功标准**: 启用硬件过滤后 CPU 使用率降低 ≥ 50%
- **注意**: 此指标需要真实硬件测试，无法通过 MockBackend 验证。软件过滤性能通过 SC-003 验证。

### SC-003: 软件过滤延迟
- **指标**: 软件过滤增加的延迟 < 10 微秒/消息
- **测量方法**: 测量过滤前后的消息处理时间
- **成功标准**: 每条消息的过滤延迟 < 10 μs

### SC-004: 资源稳定性
- **指标**: 长时间运行无内存泄漏
- **测量方法**: 运行 1 小时，每分钟采样一次内存使用量，计算峰值与初始值的比率
- **成功标准**: 峰值内存 / 初始内存 < 1.10（波动 < 10%）
- **验证工具**: valgrind (Linux) 或 miri (跨平台) 用于泄漏检测

### SC-005: 测试覆盖率
- **指标**: 新功能测试覆盖率 ≥ 90%
- **测量方法**: 使用 cargo-tarpaulin 或 llvm-cov
- **成功标准**: 覆盖率 ≥ 90%

## 关键实体

1. **CanBackendAsync**: 异步后端 trait ✅ 已在 002 中实现
2. **MessageFilter**: 消息过滤器 trait，定义 `matches()`, `priority()`, `is_hardware()` 方法
3. **FilterConfig**: 过滤器配置，支持从 TOML 加载
4. **IdFilter**: ID 过滤器实现（支持精确匹配和掩码匹配）
5. **RangeFilter**: 范围过滤器实现（匹配 ID 范围）
6. **FilterChain**: 过滤器链，管理多个过滤器，自动处理硬件/软件过滤器回退
7. **ConnectionMonitor**: 连接状态监控器
8. **QueueOverflowPolicy**: 队列溢出策略枚举（DropOldest/DropNewest/Block）

> **注意**: 硬件过滤器和软件过滤器不是独立的结构体。所有过滤器（如 IdFilter、RangeFilter）都实现 `MessageFilter` trait，通过 `is_hardware()` 方法返回值区分：
> - `is_hardware() == true`: 硬件过滤器，由 CAN 控制器执行
> - `is_hardware() == false`: 软件过滤器，由 CPU 执行
>
> FilterChain 会自动管理：优先使用硬件过滤器，超出硬件限制时回退到软件过滤。

## 边界情况

1. **异步取消**: 异步操作被取消时的资源清理
2. **多过滤器匹配**: 多个过滤器使用 OR 逻辑，任一过滤器匹配即通过（无冲突概念）
3. **硬件过滤器数量限制**: 超过硬件支持的过滤器数量时回退到软件过滤
4. **断开重连**: 硬件断开后重新连接的状态恢复
5. **队列溢出**: 按用户配置的 `QueueOverflowPolicy` 处理（默认 DropOldest）
6. **Block 策略超时**: 使用 Block 策略时超时后返回 `QueueFull` 错误
7. **内存压力**: 内存不足时减小队列大小，触发更频繁的 QueueOverflowPolicy 处理

### US1: 异步消息处理

**作为** 一名开发高性能 CAN 应用的工程师
**我希望** 使用 async/await 语法处理 CAN 消息
**以便** 我可以在不阻塞主线程的情况下处理多个 CAN 通道

#### 验收场景

**场景 1.1**: 异步发送消息
```
Given 已初始化的异步后端
When 我调用 send_message_async(&msg).await
Then 消息应该被异步发送
And 不阻塞当前任务
```

**场景 1.2**: 异步接收消息
```
Given 已初始化的异步后端
When 我调用 receive_message_async(timeout).await
Then 应该异步等待消息到达
And 超时后返回 None
```

**场景 1.3**: 多通道并发处理
```
Given 两个已打开的 CAN 通道
When 我同时在两个通道上接收消息
Then 两个通道应该独立并发处理
And 不会相互阻塞
```

**场景 1.4**: 功能开关
```
Given 项目不需要异步功能
When 我不启用 "async" feature
Then 异步 API 不会被编译
And 不会增加二进制大小
```

- **FR-002**: ✅ 异步 API 通过 `async` feature flag 启用，默认不编译，绑定 tokio 运行时
- **FR-003**: ✅ 支持可配置的接收超时
- **FR-004**: ✅ 支持多通道并发操作
