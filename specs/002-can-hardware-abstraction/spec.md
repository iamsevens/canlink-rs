# 功能规范: CAN 硬件抽象层

**功能分支**: `002-can-hardware-abstraction`
**创建时间**: 2026-01-08
**状态**: 草稿
**输入**: 用户描述: "CAN 硬件抽象层设计，支持多种硬件后端的统一接口"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 统一的硬件接口 (优先级: P1)

作为应用开发者，我需要使用统一的接口操作不同品牌的 CAN 硬件（TSMaster、PEAK、Kvaser 等），以便在不修改应用代码的情况下切换硬件供应商。

**优先级原因**: 这是抽象层的核心价值，确保应用代码与具体硬件解耦，是所有后续功能的基础。

**独立测试**: 可以通过编写使用统一接口的测试代码，分别连接 TSMaster 和 Mock 硬件，验证相同的代码在两种硬件上都能正常工作。

**验收场景**:

1. **给定** 应用代码使用统一的 CAN 接口，**当** 配置使用 TSMaster 硬件，**那么** 应用能够成功初始化并发送接收消息
2. **给定** 应用代码使用统一的 CAN 接口，**当** 配置使用 Mock 硬件（测试用），**那么** 应用能够成功初始化并发送接收消息
3. **给定** 应用已使用 TSMaster 硬件运行，**当** 切换配置为其他硬件后端，**那么** 应用无需修改代码即可使用新硬件
4. **给定** 用户查询可用的硬件后端，**当** 系统列出已注册的后端，**那么** 显示所有支持的硬件类型（TSMaster、Mock 等）

---

### 用户故事 2 - 硬件能力发现 (优先级: P2)

作为应用开发者，我需要在运行时查询硬件的能力（支持的通道数、CAN-FD 支持、最大波特率等），以便根据硬件能力调整应用行为。

**优先级原因**: 不同硬件有不同的能力，应用需要能够适配硬件限制，避免使用不支持的功能。

**独立测试**: 可以通过查询不同硬件后端的能力信息，验证返回的能力描述准确反映硬件特性。

**验收场景**:

1. **给定** 连接了 TSMaster 硬件，**当** 查询硬件能力，**那么** 返回支持的通道数、CAN-FD 支持状态、最大波特率等信息
2. **给定** 硬件不支持 CAN-FD，**当** 应用尝试发送 CAN-FD 消息，**那么** 系统在发送前检测到不支持并返回明确错误
3. **给定** 硬件只有 2 个通道，**当** 应用尝试打开第 3 个通道，**那么** 系统返回"通道不存在"错误
4. **给定** 查询硬件支持的波特率列表，**当** 应用选择波特率，**那么** 只能从支持的列表中选择

---

### 用户故事 3 - 无硬件测试支持 (优先级: P3)

作为应用开发者，我需要在没有物理硬件的情况下测试应用逻辑（使用 Mock 后端），以便在开发环境中快速迭代和自动化测试。

**优先级原因**: Mock 后端使开发和测试更加便捷，但不影响核心功能，可以在基础抽象层稳定后实现。

**独立测试**: 可以通过使用 Mock 后端运行完整的应用测试套件，验证所有功能在无硬件环境下正常工作。

**验收场景**:

1. **给定** 配置使用 Mock 后端，**当** 应用发送 CAN 消息，**那么** Mock 后端记录发送的消息供验证
2. **给定** Mock 后端配置了预设的接收消息，**当** 应用读取消息，**那么** 接收到预设的消息
3. **给定** Mock 后端配置模拟总线错误，**当** 应用运行，**那么** 触发错误处理逻辑
4. **给定** 使用 Mock 后端运行自动化测试，**当** 测试完成，**那么** 所有测试通过且无需物理硬件

### 边界情况

- 当请求的硬件后端未注册或不可用时，系统如何报告错误？
- 当硬件后端初始化失败时，系统如何提供有用的错误信息？
- 当应用尝试使用硬件不支持的功能时，系统如何在编译时或运行时检测？
- 当多个应用同时使用同一硬件时，系统如何处理资源冲突？
- 当硬件后端版本不兼容时，系统如何检测并报告？
- 当从一个硬件后端切换到另一个时，如何确保状态正确迁移或清理？

## 需求 *(必填)*

### 术语说明

本文档中使用以下术语约定：
- **硬件后端** / **后端**: 指实现 CanBackend trait 的具体硬件驱动（如 TSMaster、PEAK、Mock）。两者可互换使用，简短上下文使用"后端"，需要明确时使用"硬件后端"
- **后端注册表** / **BackendRegistry**: 中文文档使用"后端注册表"，代码和技术文档使用 `BackendRegistry`
- **抽象层** / **HAL**: 指本规范定义的硬件抽象层（Hardware Abstraction Layer）

### 功能需求

- **FR-001**: 系统必须提供统一的硬件接口定义，所有硬件后端必须实现该接口
  - **核心方法**: initialize, close, open_channel, close_channel, send_message, receive_message, get_capability
  - **详细定义**: 参见 contracts/backend-trait.md
- **FR-002**: 系统必须支持运行时注册和发现硬件后端
  - **注册机制**: 通过 BackendRegistry::register() 静态方法注册后端工厂
  - **发现机制**: 通过 BackendRegistry::list_backends() 列出所有已注册后端
  - **详细定义**: 参见 contracts/backend-registry.md
- **FR-003**: 系统必须提供硬件能力查询接口，包括：支持的通道数、CAN-FD 支持、最大波特率、支持的过滤器数量、时间戳精度
  - **说明**: 时间戳精度通过 TimestampPrecision enum 表示，作为 HardwareCapability 结构体的一个字段
- **FR-004**: 系统必须支持通过 TOML 配置文件（canlink.toml）选择使用哪个硬件后端，支持结构化配置和后端特定参数
- **FR-005**: 系统必须提供 Mock 硬件后端用于测试
- **FR-006**: 系统必须定义统一的错误类型，所有硬件后端使用相同的错误表示
  - **错误码范围**:
    - 1000-1999: 硬件相关错误（初始化失败、设备未找到、通道错误）
    - 2000-2999: 协议相关错误（无效 ID、数据长度错误、格式错误）
    - 3000-3999: 配置相关错误（配置文件错误、参数无效）
    - 4000-4999: 系统相关错误（内存不足、权限错误、超时）
- **FR-007**: 系统必须定义统一的 CAN 消息类型，独立于具体硬件
- **FR-008**: 系统必须支持硬件后端的版本管理和兼容性检查，使用语义版本控制，主版本号相同即视为兼容（如 1.x.x 与 1.y.z 兼容）
  - **检查时机**: 在 BackendRegistry::register() 注册后端时进行版本兼容性检查
  - **实现位置**: Phase 3 (T027 BackendRegistry 实现)
- **FR-009**: 系统必须提供硬件后端的生命周期管理（初始化、运行、关闭），初始化失败时自动重试（默认重试 3 次，间隔 1 秒）
  - **重试范围**: 重试逻辑由 HAL 层的 BackendRegistry 提供，作为所有后端的辅助函数。后端实现本身不需要实现重试逻辑
  - **重试配置**: 重试次数和间隔可通过 BackendConfig 配置。默认值（硬编码）为 3 次重试，每次间隔 1 秒；用户可在配置文件中覆盖这些默认值
  - **错误处理**: 重试失败后返回 `CanError::InitializationFailed`，包含：
    - 所有重试尝试的错误列表（用于诊断）
    - 最后一次失败的详细错误信息
    - 重试次数和总耗时
  - **重试策略**: 使用固定间隔重试（不使用指数退避），适合硬件初始化场景
- **FR-010**: 系统必须确保接口设计支持同步和异步操作模式，硬件后端实例要求外部同步（不提供内部锁），调用者负责多线程访问的同步控制
  - **详细策略**: 参见 research.md "线程安全模型" 章节
- **FR-011**: 系统必须提供清晰的文档，说明如何实现新的硬件后端
- **FR-012**: 系统必须支持硬件后端的插件式加载，优先使用编译时静态链接，可选支持动态库加载作为高级功能
  - **v0.1.0 范围**: 仅实现编译时静态链接（通过 Cargo workspace 和 feature flags）
  - **未来版本**: 动态库加载功能延迟到 v0.2.0 或更高版本
  - **详细说明**: 参见 plan.md "项目结构" 章节
- **FR-013**: 系统必须提供命令行工具（CLI），支持后端管理、消息收发、配置验证等操作
  - **基本命令**:
    - `canlink list` - 列出所有可用的硬件后端
    - `canlink info <backend>` - 查询指定后端的硬件能力
    - `canlink send <backend> <channel> <id> <data>` - 发送 CAN 消息
    - `canlink receive <backend> <channel>` - 接收 CAN 消息
    - `canlink validate <config-file>` - 验证配置文件
  - **输入验证规则**:
    - CAN ID: 标准帧 0x000-0x7FF (11位)，扩展帧 0x00000000-0x1FFFFFFF (29位)
    - 数据长度: CAN 2.0 为 0-8 字节，CAN-FD 为 0-64 字节
    - 通道号: 非负整数，范围由硬件能力决定
    - 数据格式: 十六进制字节序列，如 "01 02 03 04"
  - **退出码**: 0=成功，1=一般错误，2=参数错误，3=硬件错误
  - **输出格式**: 支持 JSON 格式（`--json`）和人类可读格式（默认）
  - **错误处理**: 遵循文本输入/输出协议（stdin/args → stdout, 错误 → stderr），使用标准退出码
  - **测试覆盖率**: CLI 工具包含在 SC-003 的 90% 测试覆盖率要求中，需要集成测试验证所有命令
  - **章程合规**: 满足章程原则 II（所有库必须提供 CLI 工具）
- **FR-014**: 系统必须为 CanBackend trait 的核心方法定义明确的并发行为和资源管理需求（补充 FR-001）
  - **详细定义**: 参见 contracts/backend-trait.md
  - **并发行为**: send_message 和 receive_message 方法要求外部同步，不保证内部线程安全
  - **资源清理**: close 方法必须释放所有资源（详见 contracts/backend-trait.md）
- **FR-015**: 系统必须为 BackendRegistry 定义线程安全和注册行为需求
  - **线程安全**: BackendRegistry::register() 和 list_backends() 必须是线程安全的，支持并发调用
  - **重复注册**: 重复注册同名后端时，返回 CanError::BackendAlreadyRegistered 错误，不覆盖已有注册
  - **注册顺序**: list_backends() 返回的后端列表按注册顺序排序
  - **注册失败**: 注册失败时不影响已注册的其他后端
- **FR-016**: 系统必须为配置文件定义完整的模式和错误处理需求
  - **配置模式**: canlink.toml 必须包含 [backend] 节，指定 type 字段（必需）和可选的后端特定参数
  - **示例配置**:
    ```toml
    [backend]
    type = "tscan"
    device_index = 0
    retry_count = 3
    retry_interval_ms = 1000
    ```
  - **配置验证**: 加载配置时验证所有必需字段，后端特定参数由后端实现验证
  - **配置缺失**: 配置文件不存在时返回 CanError::ConfigNotFound，使用默认配置（如果适用）
  - **配置错误**: 配置格式错误时返回 CanError::ConfigParseError，包含错误位置和原因
  - **热重载**: v0.1.0 不支持配置热重载，需要重新初始化后端
- **FR-017**: 系统必须为错误类型定义上下文信息和方法错误映射需求
  - **错误上下文**: 每个 CanError 变体必须包含：操作名称、相关参数值、时间戳（可选）
  - **方法错误映射**:
    - initialize: InitializationFailed, ConfigError, DeviceNotFound
    - open_channel: ChannelNotAvailable, ChannelAlreadyOpen, HardwareError
    - close_channel: ChannelNotOpen, HardwareError
    - send_message: InvalidMessage, ChannelNotOpen, TransmitFailed, HardwareError
    - receive_message: ChannelNotOpen, ReceiveFailed, HardwareError
    - get_capability: HardwareError
  - **错误链**: 支持错误链（cause），保留底层错误信息用于调试
- **FR-018**: 系统必须为 CAN 消息类型定义完整的验证和特殊帧处理需求
  - **ID 验证**: 标准帧 ID 必须在 0x000-0x7FF 范围内，扩展帧 ID 必须在 0x00000000-0x1FFFFFFF 范围内
  - **数据长度验证**: CAN 2.0 数据长度 0-8 字节，CAN-FD 数据长度 0-64 字节（必须是有效的 DLC 值）
  - **RTR 帧处理**: 远程帧（RTR）的数据字段必须为空（长度为 0），发送时忽略数据内容
  - **时间戳来源**: 时间戳由硬件提供（如果支持），否则由软件在接收时生成
  - **时间戳精度**: 通过 HardwareCapability::timestamp_precision 字段查询，可能值：Microsecond, Millisecond, None
- **FR-019**: 系统必须定义日志记录和调试支持需求
  - **日志级别**: 支持 Error, Warn, Info, Debug, Trace 五个级别
  - **日志内容**: 包括操作名称、参数、结果、耗时、错误信息
  - **日志框架**: 使用 Rust 标准日志框架（log crate），不强制特定实现
  - **调试支持**: 所有公共类型实现 Debug trait，提供可读的调试输出
- **FR-020**: 系统必须定义 Rust 工具链和平台支持需求
  - **MSRV**: 最低支持 Rust 1.75.0
  - **支持平台**: Windows (x64), Linux (x64), macOS (x64, ARM64)
  - **平台特定**: TSCan 后端仅支持 Windows，其他后端跨平台
  - **编译特性**: 使用 feature flags 控制可选功能和后端选择

### 关键实体

- **硬件后端接口**: 定义所有硬件必须实现的操作，包括：通道管理、消息收发、状态查询、错误处理。要求外部同步，close 方法幂等且释放所有资源
- **后端注册表**: 管理已注册的硬件后端，支持后端枚举（列出可用后端）、查询和选择。线程安全，防止重复注册，按注册顺序返回列表
- **硬件能力描述**: 描述硬件的能力和限制，包括：通道数、CAN-FD 支持、波特率范围、过滤器数量、时间戳精度（Microsecond/Millisecond/None）
- **统一消息类型**: 与硬件无关的 CAN 消息表示，包括：ID（标准/扩展）、数据（0-8/0-64字节）、时间戳、标志位（FD/BRS/ESI/RTR）。RTR 帧数据长度必须为 0
- **统一错误类型**: 与硬件无关的错误表示，包括：错误类别、错误码、错误描述、上下文信息（操作名称、参数值、时间戳）、错误链（cause）
- **配置管理**: TOML 配置文件（canlink.toml），包含 [backend] 节和 type 字段。支持后端特定参数，验证必需字段，提供明确的错误信息
- **Mock 后端**: 模拟硬件行为的测试后端，支持预设消息、错误注入、行为验证、消息记录
- **CLI 工具**: 命令行界面，提供后端管理、消息收发、配置验证等功能。支持 JSON 和人类可读输出，标准退出码
- **日志系统**: 使用 log crate，支持 5 个级别（Error/Warn/Info/Debug/Trace），记录操作、参数、结果、耗时
- **平台支持**: MSRV 1.75.0，支持 Windows/Linux/macOS (x64)，TSCan 后端仅限 Windows

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 开发者能够在 10 分钟内实现一个新的硬件后端（基于示例代码）
  - **测量方法**: 从阅读 quickstart.md 开始，到实现 CanBackend trait 的基本方法（initialize, close, send_message, receive_message）并通过编译，总时间 ≤ 10 分钟
  - **成功标准**: 新后端能够注册到 BackendRegistry 并通过基础的契约测试（发送/接收一条消息）
  - **范围说明**: "实现"指完成以下步骤：
    1. 阅读 quickstart.md 中的"实现新后端"章节（约 2 分钟）
    2. 从模板创建新的后端 crate（约 1 分钟）
    3. 实现 CanBackend trait 的 4 个基本方法（约 5 分钟）
    4. 编译通过并注册到 BackendRegistry（约 2 分钟）
  - **不包括**: 完整的错误处理、性能优化、全面的测试覆盖（这些在后续开发中完成）
  - **验证计划**: 在实施完成后，邀请 3-5 名不同经验水平的开发者（初级、中级、高级）实际测试，记录每人的完成时间。如果平均时间超过 10 分钟，分析瓶颈并改进文档或简化流程
  - **基准数据**: 待实施后更新（目标：平均时间 ≤ 10 分钟，90th 百分位 ≤ 15 分钟）
- **SC-002**: 应用代码在不同硬件后端之间切换时，无需修改任何业务逻辑代码
  - **测量方法**: 编写一个使用 CanBackend trait 的示例应用，通过修改配置文件在 Mock 后端和 LibTSCAN 后端之间切换
  - **成功标准**: 应用代码（除配置加载外）完全不变，两个后端都能正常运行并通过相同的功能测试
- **SC-003**: 使用 Mock 后端的自动化测试覆盖率达到 90% 以上
  - **测量方法**: 使用 `cargo-tarpaulin` 或 `cargo-llvm-cov` 测量代码覆盖率
  - **成功标准**: 运行 `cargo tarpaulin --workspace` 后，总体覆盖率 ≥ 90%，核心模块（backend trait、registry、error handling）覆盖率 ≥ 95%
- **SC-004**: 硬件能力查询响应时间小于 1 毫秒
  - **测量方法**: 使用 `criterion` 基准测试，调用 `get_capability()` 方法 1000 次，计算平均响应时间
  - **测试环境要求**:
    - CPU: 现代 x86_64 处理器（≥ 2.0 GHz，≥ 4 核心）
    - 内存: ≥ 8 GB RAM
    - 操作系统: Windows 10/11 或 Linux (kernel ≥ 5.0)
    - 编译模式: Release 模式（`cargo bench`）
  - **成功标准**: 平均响应时间 < 1ms，99th 百分位 < 2ms
  - **注**: 基准测试代码应在注释中记录实际测试环境规格，以便结果可重现
- **SC-005**: 抽象层的性能开销小于 5%（相比直接调用硬件 API）
  - **测量方法**: 使用 `criterion` 基准测试，对比通过抽象层发送 1000 条消息 vs 直接调用 LibTSCAN API 发送 1000 条消息的时间
  - **基准说明**: "直接调用硬件 API" 指直接使用 LibTSCAN 的 C API（通过 FFI），不经过任何 Rust 封装
  - **测试场景**: 在相同硬件和配置下，测量以下操作的时间：
    - 场景 1: 通过抽象层（CanBackend trait）发送 1000 条标准 CAN 消息
    - 场景 2: 直接调用 LibTSCAN FFI 发送 1000 条标准 CAN 消息（使用 `canlink-tscan-sys` 中的原始 FFI 绑定）
  - **实现细节**（在基准测试代码中记录）:
    - 场景 1 实现: 使用 `TSCanBackend::send_message()` 方法
    - 场景 2 实现: 直接调用 `tscan_transmit_can_async()` FFI 函数，手动构造 `TLIBCAN` 结构体
    - 消息内容: 固定 ID (0x123)，8 字节数据 (0x01-0x08)
    - 预热: 每个场景先发送 100 条消息预热，然后测量 1000 条消息的时间
  - **成功标准**: 抽象层开销 = (场景1时间 - 场景2时间) / 场景2时间 < 5%
  - **注**: 基准测试代码应包含详细注释，说明如何构造"直接调用"场景，以便其他开发者验证
- **SC-006**: 所有硬件后端使用相同的错误类型，错误处理代码可复用 100%

## 风险与缓解措施 *(新增)*

### 技术风险

- **R-001**: **性能风险** - 抽象层可能引入超过 5% 的性能开销
  - **影响**: 高频消息收发场景下性能不达标
  - **可能性**: 中等
  - **缓解措施**:
    - 使用零成本抽象设计（trait 对象避免动态分发）
    - 通过 `criterion` 基准测试持续监控性能
    - 如发现瓶颈，使用 `inline` 优化关键路径
  - **应急计划**: 如性能开销 > 5%，考虑使用泛型而非 trait 对象，或提供 unsafe 的零开销 API

- **R-002**: **线程安全风险** - 外部同步要求可能导致使用错误
  - **影响**: 用户在多线程环境下使用不当导致数据竞争
  - **可能性**: 中等
  - **缓解措施**:
    - 在文档中明确说明线程安全要求
    - 提供线程安全的包装类型（如 `Arc<Mutex<Backend>>`）作为可选方案
    - 在 quickstart.md 中提供多线程使用示例
  - **应急计划**: 如用户反馈困难，考虑在 v0.2.0 中提供内部同步的 `SyncBackend` 包装器

- **R-003**: **后端兼容性风险** - 不同硬件后端的行为差异可能导致抽象层设计不足
  - **影响**: 某些后端无法完全实现 CanBackend trait
  - **可能性**: 低
  - **缓解措施**:
    - 优先实现 LibTSCAN 和 Mock 后端验证设计
    - 在 trait 设计中预留扩展点（如 `backend_specific_config`）
    - 通过契约测试确保所有后端行为一致
  - **应急计划**: 如发现设计缺陷，通过 trait 方法添加（保持向后兼容）或在 v0.2.0 中引入 breaking change

- **R-004**: **FFI 绑定风险** - C/C++ FFI 绑定可能引入内存安全问题
  - **影响**: 内存泄漏、段错误、未定义行为
  - **可能性**: 中等
  - **缓解措施**:
    - 使用 `bindgen` 自动生成绑定
    - 在 unsafe 代码周围添加安全包装
    - 使用 `valgrind` 或 `miri` 检测内存问题
    - 编写全面的测试覆盖 FFI 边界
  - **应急计划**: 如发现严重内存问题，暂停发布并进行安全审计

### 项目风险

- **R-005**: **依赖风险** - LibTSCAN 库可能存在 bug 或版本不兼容
  - **影响**: 后端功能异常或无法使用
  - **可能性**: 低
  - **缓解措施**:
    - 在 README 中明确支持的 LibTSCAN 版本
    - 提供 Mock 后端作为备选方案
    - 与同星保持沟通，及时反馈问题
  - **应急计划**: 如 LibTSCAN 存在严重问题，优先实现 TSMaster 后端或 SocketCAN 后端

- **R-006**: **文档风险** - 文档不足可能导致用户难以使用
  - **影响**: 用户采用率低，支持成本高
  - **可能性**: 低（已有完善的文档计划）
  - **缓解措施**:
    - SC-007 确保 100% 文档覆盖
    - quickstart.md 提供快速入门指南
    - 提供多个示例代码
  - **应急计划**: 根据用户反馈持续改进文档

- **R-007**: **范围蔓延风险** - 用户可能要求超出 v0.1.0 范围的功能
  - **影响**: 延迟发布，增加复杂度
  - **可能性**: 中等
  - **缓解措施**:
    - 明确定义范围内和范围外功能
    - 使用 ROADMAP.md 追踪未来功能
    - 通过 feature flags 支持可选功能
  - **应急计划**: 坚持 MVP 原则，将额外功能推迟到 v0.2.0

## Clarifications

### Session 2026-01-08

- Q: 当多个线程同时访问同一个硬件后端实例时，抽象层应该如何处理并发？ → A: 要求外部同步 - 抽象层不提供内部锁，调用者负责同步（推荐用于性能）
- Q: 硬件后端的配置（选择使用哪个后端）应该通过什么方式提供？ → A: TOML 配置文件 - 使用 canlink.toml 配置文件，支持结构化配置和多个参数
- Q: 硬件后端的插件加载机制应该优先使用哪种方式？ → A: 优先静态链接 - 默认编译时静态链接后端，可选支持动态加载
- Q: 当硬件后端初始化失败时，系统应该如何处理？ → A: 自动重试机制 - 系统内置重试逻辑（如重试 3 次，间隔 1 秒），失败后才返回错误
- Q: 当硬件后端版本与抽象层版本不同时，系统应该如何判断兼容性？ → A: 主版本号兼容 - 只要主版本号相同就允许加载（如 1.x.x 与 1.y.z 兼容），遵循语义版本控制标准

## 架构说明 *(特殊)*

此规范定义的抽象层将作为所有硬件后端的基础：

```
应用层
   ↓
CAN 硬件抽象层 (002) ← 本规范
   ↓
┌──────────────┬──────────────┬────────┐
│ TSMaster (001)│ PEAK (future)│ Mock   │
└──────────────┴──────────────┴────────┘
```

**实现顺序**：
1. 定义抽象层接口（trait）
2. 实现 Mock 后端（用于验证接口设计）
3. 将 TSMaster 实现迁移到抽象层接口
4. 未来添加其他硬件后端
