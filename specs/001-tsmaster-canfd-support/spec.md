# 功能规范: TSMaster CAN/CAN-FD 总线收发支持

**功能分支**: `001-tsmaster-canfd-support`
**创建时间**: 2026-01-08
**状态**: 草稿
**输入**: 用户描述: "支持同星的TSMaster工具进行CAN/CAN-FD总线的收发"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 基础 CAN 消息收发 (优先级: P1)

作为汽车电子工程师，我需要通过 TSMaster 硬件发送和接收标准 CAN 2.0 消息，以便进行基本的总线通信测试和诊断。

**优先级原因**: 这是最基础的功能，是所有后续功能的前提。标准 CAN 2.0 是最广泛使用的协议，必须首先支持。

**独立测试**: 可以通过连接 TSMaster 硬件设备，发送一条标准 CAN 消息并接收回显，验证基本收发功能正常工作。

**验收场景**:

1. **给定** TSMaster 硬件已连接并初始化，**当** 用户发送一条标准 CAN 数据帧（ID: 0x123, 数据: [0x01, 0x02, 0x03]），**那么** 消息成功发送到总线上
2. **给定** TSMaster 硬件正在监听总线，**当** 总线上有 CAN 消息传输，**那么** 系统能够接收并解析该消息的 ID 和数据
3. **给定** 用户配置了波特率为 500kbps，**当** 初始化 CAN 通道，**那么** 硬件按指定波特率工作
4. **给定** 发送队列中有多条消息，**当** 执行批量发送，**那么** 所有消息按顺序发送成功
5. **给定** 用户需要请求数据，**当** 发送远程帧（RTR 标志置位），**那么** 远程帧成功发送且接收方能识别 RTR 标志
6. **给定** 总线上发生错误，**当** 查询控制器状态，**那么** 系统正确报告当前状态（Error Active、Error Passive 或 Bus-Off）

---

### 用户故事 2 - CAN-FD 高速数据传输 (优先级: P2)

作为需要高带宽数据传输的工程师，我需要使用 CAN-FD 协议发送和接收大数据包（最多 64 字节），以便进行高速数据采集和固件更新。

**优先级原因**: CAN-FD 是现代汽车网络的标准，支持更高的数据速率和更大的数据负载，是项目的核心价值之一。

**独立测试**: 可以通过发送一条 64 字节的 CAN-FD 消息，并验证接收端正确接收所有数据，测试 CAN-FD 功能独立工作。

**验收场景**:

1. **给定** TSMaster 硬件支持 CAN-FD，**当** 用户发送一条 CAN-FD 消息（64 字节数据），**那么** 消息以 CAN-FD 格式成功发送
2. **给定** 配置了 CAN-FD 仲裁段波特率 500kbps 和数据段波特率 2Mbps，**当** 发送 CAN-FD 消息，**那么** 硬件按配置的双波特率工作
3. **给定** 接收到 CAN-FD 消息，**当** 解析消息，**那么** 系统正确识别 FD 标志、BRS 标志和 ESI 标志
4. **给定** 混合 CAN 2.0 和 CAN-FD 消息在总线上传输，**当** 系统接收消息，**那么** 能够正确区分和处理两种协议的消息

---

### 用户故事 3 - 多通道并发操作 (优先级: P3)

作为需要同时监控多个 CAN 网络的工程师，我需要同时操作多个 CAN 通道（例如：动力总成网络和车身网络），以便进行复杂的系统集成测试。

**优先级原因**: 现代车辆有多个 CAN 网络，支持多通道是完整解决方案的必要功能，但可以在基础功能稳定后再实现。

**独立测试**: 可以通过同时在两个不同的 CAN 通道上发送和接收消息，验证多通道功能独立工作且互不干扰。

**验收场景**:

1. **给定** TSMaster 硬件有多个 CAN 通道，**当** 用户同时初始化通道 1 和通道 2，**那么** 两个通道都成功初始化
2. **给定** 两个通道都在运行，**当** 在通道 1 发送消息同时在通道 2 接收消息，**那么** 两个操作互不干扰且都成功完成
3. **给定** 每个通道配置了不同的波特率，**当** 消息在各自通道传输，**那么** 每个通道按各自配置的波特率工作
4. **给定** 一个通道发生错误，**当** 处理错误，**那么** 其他通道继续正常工作不受影响

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须支持通过 TSMaster 硬件发送标准 CAN 2.0A（11 位 ID）和 CAN 2.0B（29 位扩展 ID）数据帧
- **FR-002**: 系统必须支持通过 TSMaster 硬件接收标准 CAN 2.0A 和 CAN 2.0B 数据帧
- **FR-003**: 系统必须支持发送和接收 CAN 远程帧（RTR），用于请求数据
- **FR-004**: 系统必须支持 CAN-FD 协议，包括最多 64 字节的数据负载
- **FR-005**: 系统必须支持配置 CAN 波特率（常见值：125kbps, 250kbps, 500kbps, 1Mbps）
- **FR-006**: 系统必须支持配置 CAN-FD 的仲裁段和数据段波特率（数据段最高 5Mbps）
- **FR-007**: 系统必须能够检测 TSMaster 硬件的连接状态
- **FR-008**: 系统必须能够初始化和关闭 CAN 通道
- **FR-009**: 系统必须支持设置 CAN 消息过滤器以减少不必要的接收
- **FR-010**: 系统必须提供消息时间戳，精度至少为微秒级
- **FR-011**: 系统必须能够检测和报告 CAN 总线错误类型，包括：位错误（Bit Error）、填充错误（Stuff Error）、CRC 错误（CRC Error）、ACK 错误（ACK Error）、格式错误（Form Error）
- **FR-012**: 系统必须能够查询和报告 CAN 控制器状态，包括：Error Active（错误主动）、Error Passive（错误被动）、Bus-Off（总线关闭）
- **FR-013**: 系统必须提供错误计数器信息（发送错误计数 TEC 和接收错误计数 REC）
- **FR-014**: 系统必须支持同时操作多个 CAN 通道（如果硬件支持）
- **FR-015**: 系统必须提供发送确认机制，指示消息是否成功发送
- **FR-016**: 系统必须支持周期性消息发送（按固定时间间隔自动发送）
- **FR-017**: 系统必须能够查询 TSMaster 硬件的能力（支持的通道数、CAN-FD 支持等）
- **FR-018**: 系统必须支持 Bus-Off 状态的自动或手动恢复

### 关键实体

- **CAN 消息**: 表示一条 CAN 或 CAN-FD 消息，包含 ID（标准或扩展）、数据长度码（DLC）、数据字节、时间戳、标志位（RTR, FD, BRS, ESI）
- **CAN 通道**: 表示一个物理 CAN 接口，包含通道索引、波特率配置、状态（已初始化、运行中、错误）、统计信息（发送/接收计数、错误计数）
- **硬件设备**: 表示 TSMaster 硬件设备，包含设备序列号、固件版本、支持的通道列表、能力标志
- **消息过滤器**: 表示接收过滤规则，包含 ID 掩码、ID 匹配值、过滤器类型（接受/拒绝）
- **控制器状态**: 表示 CAN 控制器的当前状态，包含状态类型（Error Active/Passive/Bus-Off）、发送错误计数（TEC）、接收错误计数（REC）
- **总线错误**: 表示检测到的总线错误事件，包含错误类型（位错误、填充错误、CRC 错误、ACK 错误、格式错误）、发生时间戳、错误位置

- **SC-008**: 总线错误恢复时间小于 100 毫秒，确保快速恢复通信
- **SC-009**: 系统能够在 50 毫秒内检测并报告总线错误类型（位错误、填充错误等）

- 假设用户具备基本的 CAN 总线知识（了解 ID、DLC、波特率等概念）
