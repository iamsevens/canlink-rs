# 规范 006: CLI 生产能力收口与模拟后端解耦

**功能分支**: `006-cli-production-boundary`
**创建时间**: 2026-03-22
**状态**: 草稿
**输入**: 用户描述: "`canlink-cli` 面向真实硬件用户，不应深度绑定 `mock`；主 CLI 仅保留真实使用路径，并同步收缩对外发布面"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 真实硬件用户的核心 CLI 路径 (优先级: P1)

作为使用真实 CAN 硬件的 Rust 工程师，我需要 `canlink-cli` 只暴露真实设备使用必需的命令，以便我能够稳定地完成后端发现、能力查询、消息发送、消息接收和配置校验，而不会被开发/模拟命令干扰。

**优先级原因**: 这是 `canlink-cli` 的对外定位重定义，决定了产品边界、文档口径和发布面。

**独立测试**: 在不依赖模拟后端的前提下构建并运行 CLI，验证帮助信息、命令集合和真实后端路径均符合预期。

**验收场景**:

1. **给定** 已构建的正式 CLI，**当** 用户查看帮助信息，**那么** CLI 仅展示 `list`、`info`、`send`、`receive`、`validate` 五类正式命令
2. **给定** 已注册的真实硬件后端，**当** 用户执行 `list` 或 `info`，**那么** CLI 仅展示和查询正式支持的真实后端
3. **给定** 真实硬件可用，**当** 用户执行 `send` 或 `receive`，**那么** CLI 通过真实后端完成消息发送与接收
4. **给定** 用户提供配置文件，**当** 执行 `validate`，**那么** CLI 仅对配置内容做校验，不要求模拟后端参与运行时注册
5. **给定** 用户尝试使用已移出的命令，**当** CLI 解析命令行，**那么** 应明确提示该命令不是正式 CLI 命令集合的一部分

---

### 用户故事 2 - 开发与测试能力和正式 CLI 脱钩 (优先级: P1)

作为仓库维护者，我需要继续保留模拟后端用于测试、示例和开发调试，但不把它作为正式 CLI 的默认运行时依赖，以便开发便利性和对外产品边界可以同时成立。

**优先级原因**: 如果模拟后端仍然深度绑定主 CLI，发布边界、依赖图和用户认知都会持续混乱。

**独立测试**: 在 workspace 内继续运行依赖模拟后端的测试与示例，同时验证正式 CLI 的打包与发布不再要求模拟后端进入主发布链。

**验收场景**:

1. **给定** workspace 内部测试与示例，**当** 维护者执行测试或开发验证，**那么** 模拟后端仍可继续用于这些内部场景
2. **给定** 正式 CLI 的构建与打包流程，**当** 执行发布验证，**那么** 不再要求模拟后端作为主 CLI 的正式运行时依赖
3. **给定** 对外文档，**当** 用户阅读 CLI 使用说明，**那么** 不会把模拟后端误解为正式 CLI 的默认能力
4. **给定** 模拟后端仍存在于仓库中，**当** 维护者查看发布清单，**那么** 模拟后端被识别为开发/测试能力，而不是正式发布组成部分

---

### 用户故事 3 - 发布面与文档口径一致收口 (优先级: P1)

作为发布维护者，我需要正式发布集合、发布顺序、文档说明和 CI 检查全部与新的产品边界一致，以便对外发布时不会出现“代码、文档、发布流程三套口径”。

**优先级原因**: 这次调整不仅是代码改动，更是对外产品面与发布资产的统一收口。

**独立测试**: 检查发布脚本、发布工作流、发布清单、README 与打包验证结果，确认它们都围绕同一组正式 crate 和同一条发布顺序运行。

**验收场景**:

1. **给定** 正式发布清单，**当** 维护者查看发布集合，**那么** 仅包含 `canlink-hal`、`canlink-tscan-sys`、`canlink-tscan`、`canlink-cli`
2. **给定** 发布顺序说明，**当** 维护者执行发布流程，**那么** 顺序固定为 `canlink-hal -> canlink-tscan-sys -> canlink-tscan -> canlink-cli`
3. **给定** 对外 README、CLI 文档和发布文档，**当** 用户阅读说明，**那么** 文档中的示例与命令集合只反映正式支持范围
4. **给定** CI 与发布前检查，**当** 执行质量门禁和发布 dry-run，**那么** 结果与新的 4-crate 发布面一致

---

### 边界情况

- 当当前环境中没有任何真实后端可用时，CLI 应明确报告“无可用正式后端”，而不是回退或暗示使用模拟后端
- 当用户仍然依据旧文档尝试 `isotp`、`filter`、`monitor` 命令时，CLI 应返回明确的未知命令/不支持提示
- 当仓库内部仍保留模拟后端测试时，正式对外文档不能再把这些内部路径宣传为默认用户路径
- 当发布资产仍残留 `canlink-mock`、旧命令或旧发布顺序时，发布前检查必须能够发现不一致
- 当未来新增新的真实硬件后端时，本规范不阻止扩展，但新增后端必须以“正式后端”身份显式进入文档和发布面

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须将正式 CLI 命令集合收口为 `list`、`info`、`send`、`receive`、`validate`
- **FR-002**: 系统必须将 `isotp`、`filter`、`monitor` 从正式 CLI 命令集合中移除
- **FR-003**: 系统必须确保正式 CLI 默认只暴露正式支持的真实后端，不默认暴露模拟后端
- **FR-004**: 系统必须允许模拟后端继续保留在 workspace 内，用于测试、示例和开发调试
- **FR-005**: 系统必须确保正式 CLI 的构建、打包与发布流程不再把模拟后端视为正式运行时依赖
- **FR-006**: 系统必须将对外文档中的 CLI 示例切换到正式支持的真实后端路径
- **FR-007**: 系统必须将模拟后端在文档中的定位明确为开发/测试用途，而非正式 CLI 默认能力
- **FR-008**: 系统必须将正式发布集合收口为 `canlink-hal`、`canlink-tscan-sys`、`canlink-tscan`、`canlink-cli`
- **FR-009**: 系统必须将正式发布顺序固定为 `canlink-hal -> canlink-tscan-sys -> canlink-tscan -> canlink-cli`
- **FR-010**: 系统必须确保发布脚本、发布工作流、发布清单和 README 与新的 4-crate 发布面一致
- **FR-011**: 系统必须在质量门禁中持续校验新的正式发布面和文档覆盖要求，避免后续回退
- **FR-012**: 系统必须避免对外文档、交叉链接或发布说明继续暗示 `canlink-mock` 是正式公开发布链的一部分

### 关键实体

- **正式 CLI 能力面**: 对外公开支持的用户命令集合，仅包含真实硬件用户路径
- **开发/测试能力面**: 供维护者使用的模拟、测试、示例和调试路径，不属于正式 CLI 默认能力
- **正式后端**: 当前对外正式支持并在 CLI 中暴露的真实硬件后端
- **模拟后端**: 保留在 workspace 中用于测试与开发的内部后端能力
- **正式发布集合**: 需要进入对外发布文档、发布顺序和发布脚本的 crate 集合

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: `canlink --help` 中仅包含 `list`、`info`、`send`、`receive`、`validate` 五类正式命令
- **SC-002**: 正式 CLI 的打包/发布验证不再要求 `canlink-mock` 进入主发布链
- **SC-003**: 所有发布资产都一致展示 4 个正式 crate，且顺序统一为 `canlink-hal -> canlink-tscan-sys -> canlink-tscan -> canlink-cli`
- **SC-004**: 对外 CLI 文档中不再使用 `mock` 作为默认用户示例后端，也不再宣传已移出的命令
- **SC-005**: workspace 内部依赖模拟后端的测试/开发路径仍然能够继续使用，不因正式 CLI 收口而被整体移除
- **SC-006**: 发布前文档覆盖与质量门禁在新发布面下继续通过，不因收口导致发布质量下降

## 假设 *(如适用)*

- 假设当前唯一正式对外的真实硬件后端仍是基于 LibTSCAN 的后端路径
- 假设模拟后端对测试、示例和开发调试仍有价值，因此不应整体删除
- 假设未来可以继续接入新的真实硬件后端，但本规范不要求本轮实现新的正式后端
- 假设这次收口以“对外用户路径清晰”为第一优先级，而不是保留现有 CLI 命令数量

## 范围外 *(如适用)*

- 不在本规范中实现新的真实硬件后端
- 不在本规范中设计新的插件化后端发现框架
- 不在本规范中新增单独的开发者 CLI 工具；如未来需要，可作为后续规范单独推进
- 不在本规范中删除 `canlink-mock` crate 本身；本轮仅调整其产品定位与发布角色
- 不在本规范中重新实现 `isotp`、`filter`、`monitor` 的真实硬件版 CLI 能力

## 依赖关系 *(如适用)*

- 依赖当前已经存在的正式真实后端路径与发布资产
- 依赖现有 CI、release dry-run、release publish 与文档覆盖门禁继续作为发布质量基础
- 依赖 workspace 内部测试继续可使用模拟后端验证非硬件路径
