# 规范 007: 发布前厂商库分发边界与守卫

**功能分支**: `codex/cli-boundary-006`
**创建时间**: 2026-03-24
**状态**: 已确认
**输入**: 用户要求明确发布边界，禁止随意分发 LibTSCAN DLL/Lib，并在 GitHub Actions 与本地发布流程中加守卫。

## 用户场景与测试 (必填)

### 用户故事 1 - 发布过程防止误分发厂商库 (优先级: P1)

作为发布维护者，我需要在发布流程中自动检测并阻止厂商库文件被打包或提交，以避免违规分发与后续风险。

**独立测试**:
- 运行 `python scripts/guard_vendor_bundle.py`，当仓库或包中存在 `libTSCAN.*` 时应失败。

**验收场景**:
1. **给定** 仓库中误加入 `libTSCAN.dll`，**当** 运行发布守卫，**那么** 守卫应失败并提示具体路径。
2. **给定** 打包产物中包含 `libTSCAN.lib`，**当** 运行发布守卫，**那么** 守卫应失败并提示具体包名。

---

### 用户故事 2 - 不影响日常开发/测试 (优先级: P1)

作为开发者，我需要守卫只在发布相关流程运行，而不会影响日常 `cargo build/test`，以便日常开发保持轻量。

**独立测试**:
- 常规 CI (`.github/workflows/ci.yml`) 不引入守卫步骤。

**验收场景**:
1. **给定** 日常 `cargo test`，**当** 无发布动作时，**那么** 不触发守卫脚本。
2. **给定** 仅发布工作流触发时，**当** 运行 `Release Dry Run` 或 `Release Publish`，**那么** 守卫必须先执行。

---

### 用户故事 3 - 文档明确可发布与不可发布内容 (优先级: P1)

作为对外发布负责人，我需要文档明确哪些内容可公开发布、哪些必须保留本地，避免团队成员误操作。

**独立测试**:
- 发布指南、快速指南、最终检查清单中均写明厂商库分发边界。

**验收场景**:
1. **给定** 发布文档，**当** 维护者查阅，**那么** 能清晰看到厂商库禁止分发与守卫要求。

## 需求 (必填)

### 功能需求

- **FR-001**: 发布守卫必须检查 git 追踪文件中是否存在 `libTSCAN.*`。
- **FR-002**: 发布守卫必须检查打包产物中是否存在 `libTSCAN.*`。
- **FR-003**: 发布守卫必须接入 GitHub Actions 的发布工作流。
- **FR-004**: 发布守卫必须接入本地 `scripts/release.bat` 与 `scripts/release.sh`。
- **FR-005**: 发布文档必须明确厂商库分发边界与守卫要求。

### 非功能需求

- **NFR-001**: 守卫脚本必须跨平台可运行（Windows/Linux/macOS）。
- **NFR-002**: 守卫脚本不得依赖网络访问。

## 非目标

- 不包含厂商授权申请或授权流程。
- 不修改 `canlink-tscan` 的运行时加载机制。
- 不将任何厂商 DLL/Lib 打包进仓库或发布包。

## 设计备注

- 仅在发布相关流程运行守卫，避免影响日常开发与 CI。
- 守卫检查范围包含 git 追踪文件与 crate 打包产物。
